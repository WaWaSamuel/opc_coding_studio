F-A.1 / A.2 / A.3 入口层 Gateway + 飞书 + Web

  • 入口层在编排器之外、之上,负责"渠道接入 + 协议转换 + 会话路由",不含业务逻辑。
  • 各渠道消息归一为统一 HostCommand,屏蔽渠道差异。
  • 飞书适配器:事件回调(@/私聊)+ 签名校验;回复用消息卡片 + 流式更新;need_decision 用交互按钮卡片。
  • Web 适配器:复用 React + SSE/WebSocket。
  • 身份校验来源为 Host;每对话映射 session_id 关联 task_id,支持多轮。
HostCommand = { "channel":"lark|web", "session_id":"...", "host_verified":true,
  "intent":"runtime|edit", "text":"自然语言指令",
  "messages":[{ "role":"host|assistant", "text":"...", "ts":"..." }],  // F-A.9 多轮上下文
  "attachments":[{ "type":"image", "mime":"image/png", "url|b64":"...", "name":"..." }],  // F-A.10 图片
  "reply_to":"event_id|decision_id" }

F-A.4 / A.6 流转留痕看板(hover 详情) / 成本仪表

  • 编排器关键节点推事件(role_start/thinking/tool_call/artifact/handoff/rework/error/need_decision/done),界面订阅渲染;每条事件已带统一信封 tokens{in,out} 与 latency_ms。
  • hover 展开详情(本次新增):留痕条目默认一行简述(角色 + 事件中文名 + 关键字段);鼠标 hover 弹 EventDetailPopover,展开:① 做了什么(事件语义 + payload 摘要:tool 名/入参摘要、artifact 文件清单、handoff from→to、verdict/note);② 耗时 latency_ms;③ 消耗 tokens tokens.in / out / 合计;④ 时间戳 ts。详情数据全来自既有信封字段,无需新增后端字段,仅前端 EventTimeline 把当前丢弃的 tokens/latency_ms 渲染出来。
  • 成本仪表读流转留痕的 tokens/latency_ms 聚合,按 task/角色展示;与 hover 详情同源。

F-A.5 TODO Plan 进度视图(双系统)

  • 读 State.todo_plan,渲染勾选进度;回退时高亮 status=rework 项。
  • Edit 视图同样渲染(本次新增):原 TodoView 只认 Runtime 的 dev_plan 事件;扩展为同时认 Edit 的 edit_locate 事件——Edit 部长定位阶段产出"待改文件/改造点清单"即视为 Edit 的 TODO Plan。App.tsx 的 Edit 视图区挂载 <TodoView>,与 Runtime 视图口径一致,Host 在两系统都能看到"接下来要做哪些事 / 哪些已完成"。

F-A.7 人在环决策(对话式 + IM 双通道并存)

  • 弃用全屏弹窗 DecisionModal,改为两个通道同时生效,任一回灌即收口:
    • 通道① Web 对话式回复:need_decision 事件在对话流内渲染为一条带「通过 / 打回 / 终止 + 补充意见」内联操作的 assistant 气泡(InlineDecision),Host 直接在对话里点选或文字回复;文字回复经入口层 classify_decision 解析为 verdict。
    • 通道② IM 提醒:复用已实现的飞书交互卡片(lark_adapter._render_card 的 need_decision 分支),推送到 Host 飞书,卡片按钮回调等价 POST /decision。
  • 并发去重:同一 task_id 的决策由 DecisionGate 收口(已存在),先到先得;一通道回灌后,另一通道的卡片/内联控件置为「已由其他渠道处理」灰态,避免重复回灌(幂等键 = task_id + decision 轮次)。
  • 决策回灌后编排器线程从 DecisionGate.wait 续跑(沿用 F-A.7 现有阻塞-唤醒机制),decision_timeout 仍兜底。

F-A.8 Edit 工作流可视化(reactflow 全景)

  • 触发条件:回归测试官闸门内,当本次改动 _restart_scope(files) 命中 backend/frontend/both(即改了可运行代码,而非纯角色 YAML/提示词)时,在机械回归通过后、提 PR 前,追加一段「隔离实例验证」子步骤。纯提示词/编排 YAML 改动 Runtime 下次调用即生效,不触发隔离实例(省资源)。

F-E.9 概念级定向改造可靠性加固

缺口由来(终极形态自举的关键瓶颈)终极形态要求"对话式表明需求 → Edit 工作流自主完成 runtime/web/后端全项目改造"。当前 EditWorkspace.apply_search_replace 已做仓库接地 + 精确 search/replace,但当工程师产出的 find 锚点在真实文件中找不到时,仅记 failed 留痕、不落盘——这是"改不动"的主因之一,直接拉低 Edit 自主完成全项目改造的一次成功率,卡住自举闭环。
  • 失败回灌重定位(核心):graph_edit.py 工程师步落盘后,若 SearchReplaceResult.failed 非空(锚点未命中/文件不存在),不直接 reject,而是把失败项 + 目标文件真实内容(read_targets)回灌给工程师角色重做一次——让模型基于"真实存在的代码"重新产出可命中的 search/replace,而非凭脑补的片段反复落空。重定位次数受 LoopController 上限约束(≤2),超限才判失败。
  • 接地前置强化:工程师产 diff 前,部长定位阶段(edit_locate)的 read_targets 命中文件内容默认整体进上下文(受 max_bytes 截断),并显式提示工程师"find 必须逐字摘自下方真实文件内容,不得改写/省略空白",从源头降低锚点不命中率。
  • 容错匹配兜底(可选,稳妥优先):锚点精确匹配失败时,尝试空白归一(行首尾空白、连续空格折叠)再匹配一次;仍不命中才计 failed。归一匹配命中需在留痕标注"经空白归一命中",可审计;绝不做模糊/相似度猜测改写,避免误伤。
  • 留痕透明:每次重定位、归一命中、最终失败都发 edit_engineer 子事件落库,Host 可在留痕看板看到"第几次重定位、为何失败",而非黑盒。该加固只提升成功率,不放松"锚点不命中绝不盲改"的保命底线。

5.1 技术模块详细设计

系统拆为 12 个核心模块,统一编号 M-编号。每个模块标注:职责、关键类/接口、依赖、输入输出、失败与边界处理。

M01 Gateway 入口层

职责:多渠道接入(飞书/Web)、协议归一为 HostCommand、会话路由、身份(Host)校验、意图初判(runtime/edit)。

关键类:ChannelAdapter(抽象)→ LarkAdapter / WebAdapter;SessionRouter;HostCommand(DTO);IntentClassifier(轻量规则+小模型)。

数据流:渠道事件 →(签名校验/去重)→ 归一 HostCommand →(session_id↔task_id 映射)→ 投递 Orchestrator。

边界:飞书事件需做 x-lark-signature 校验 + event_id 幂等去重;非 Host 来源直接拒绝;意图模糊触发澄清子流程(G9 预留)。

失败:渠道回调失败→本地重投队列;归一失败→回渠道提示格式错误,不进编排。

M02 Orchestrator 编排内核

职责:持有 Runtime / Edit 两张 StateGraph;按 Artifact 字段走条件边;管理 Loop/并行/回退;驱动节点=角色调用。

关键类:GraphBuilder(装配节点/边)、RuntimeGraph/EditGraph、NodeRunner(包裹一次无状态调用)、EdgeRouter(读 status→下一跳)、CompanyState(TypedDict,含 transition 字段记录"为何流转至此",借 Claude Code)。

数据流:State 入图 → NodeRunner 组装上下文 → ModelAdapter 调用 → 回写 Artifact → EdgeRouter 判定(pass/reject/need_rework/error)→ 下一节点 / 回退 / need_decision。

边界:并行节点(前后端)需在汇合节点 join;条件边必须覆盖所有 status 取值,缺省落 error 兜底节点。

失败:节点异常→交 M07 重试;超 max_iterations→置 need_decision 等 Host;图无可达边→error 上报。

M03 Role / Agent 注册层

职责:角色定义(提示词三层 + 可用 Tool/Skill + 模型档位 + 输出 schema)统一注册,供编排器按 current_role 取用。

关键类:RoleRegistry、RoleSpec{ role_id, model_tier, prompt_layers, tools[], output_schema }、PromptComposer(三层拼装)。

数据流:角色定义从 main 分支 YAML/JSON 加载 → RoleRegistry 内存索引 → NodeRunner 取 RoleSpec → PromptComposer 组装。

边界:提示词三层(common/role/task)解耦,改 common 影响全局须经 Edit 回归;model_tier 决定走小/大模型。

M04 Model Adapter 模型适配

职责:屏蔽模型差异,统一 invoke(messages, schema, tier) → Artifact;强制 JSON 输出;统计 tokens/latency;支持角色异构路由。

关键类:ModelAdapter(抽象)→ ArkAdapter(豆包,OpenAI 兼容);InvokeResult{content, tokens, latency_ms};SchemaEnforcer(response_format)。

数据流:messages →(注入 response_format)→ Ark API → 解析 JSON →(失败回 M07 自修复)→ 结构化 Artifact。

边界:密钥仅 ARK_* 环境变量;切模型只换 Adapter 实现;tier=small/large 映射不同 endpoint。

失败:JSON 解析失败→把错误回喂模型自修复(≤3);API 5xx→指数退避重试(计入 retry_counters)。

M05 Context & Memory 上下文记忆

职责:三层记忆(工作/任务/长期)读写;三级上下文流水线压缩;Runtime/Edit 物理隔离;检索 Top-K 注入;稳定前缀缓存。

关键类:MemoryManager、WorkingMem/TaskMem/LongTermMem、ResultBudgeter(Tier1)、MicroCompactor(Tier2)、Compactor(Tier3 LLM 摘要+指针)、Retriever(sqlite-vec 向量 + 关键词回退 Top-K;向量来自 Ark embedding 接口)。

三级流水线(借 Claude Code,每次调用前依次执行):① 结果预算:大产出落 DB,prompt 内留 ≤2KB 预览 + 指针(按需 fetch_artifact 取回);② 微压缩:按来源选择性清旧结果,以缓存过期为触发;③ 全量压缩:超阈值时一次 LLM 结构化九段摘要(含 Host 原话逐字引用),替换原文但保留可追溯指针。

数据流(缓存边界):公共层 → 角色层 → 长期记忆 Top-K ┊CACHE_BOUNDARY┊ 任务记忆(必要时已压缩)→ 上一棒 Artifact;边界前稳定,命中 prompt caching。

边界:Compact 保留指向 DB 完整 Artifact 的指针(可追溯,非截断);Runtime/Edit 两库两命名空间。

失败:检索为空→降级仅用任务记忆;Compact 连续失败计入熔断(M07),超限保守保留原文并告警。

M06 Tool / Skill 执行层

职责:统一工具注册与最小授权调用(fs/code.build/git.*/test.run/lark.* 等);结果按固定 schema 回写 State。

关键类:ToolRegistry、Tool{name, scopes, schema, handler}、ToolInvoker(鉴权+执行+留痕)、DangerDetector(规则/AST 危险动作检测)。

四层授权流(借 Claude Code,补 G5):① 配置规则(角色↔Tool 白名单,最快)→ ② Hook(可扩展校验)→ ③ 规则/AST 危险动作检测(fs 越界写、git reset --hard、删表等)→ ④ Host 二次确认(兜底)。危险动作"警告但不阻断,需显式确认"。

数据流:角色请求 Tool → ToolInvoker 四层校验 → 执行 → 结构化结果 → 审计留痕 → 回 State。

边界:QA 角色无 fs.write/code.build;危险操作二次确认本期以 hook 占位,门禁后启用;git.* 走 GITHUB_TOKEN。

M07 Reliability 可靠性(双重试 + 全局熔断)

职责:实现两类独立机制:① 业务回退 loop_counters(质量/跨节点/超限等 Host);② 节点重试 retry_counters(schema/JSON/Tool/超时/原地)。再叠加全局熔断常量表覆盖所有自动恢复路径。

关键类:LoopController(业务回退)、RetryController(节点重试)、SelfRepair(JSON 回喂修复)、CircuitBreakers(全局熔断)。

全局熔断常量(借 Claude Code circuit breaker,防连环失败烧 token):

MAX_COMPACT_FAILURES   = 3   # 连续压缩失败→停压缩报 Host
MAX_TOKEN_REFRESH      = 3   # 鉴权刷新失败上限
MAX_API_OVERLOAD_RETRY = 3   # 模型 5xx/限流重试
MAX_POLL_FAILURES      = 5   # 渠道轮询失败
SOFT_TASK_TOKENS       = ...  # 单任务软限→告警+降档小模型(F-D.6)
HARD_TASK_TOKENS       = ...  # 单任务硬限→即刻终止任务报 Host(F-D.6,P0/M1)
MAX_DAILY_TOKENS       = ...  # 全局日限→熔断当日新任务(F-D.6)

硬约束:节点重试到上限 不自动转 业务回退;两个计数器互不共用;每条自动恢复路径必须有对应熔断上限。

失败:loop 超 3→need_decision;retry 超 3→error 上报/降级;熔断触发→停该恢复路径并告警 Host。

M08 Persistence 持久化

职责:SQLite(./data/app.db)落 State/Checkpoint/Artifact/日志/测试集/记忆;接口抽象可换 PostgreSQL。

关键类:Repository(抽象)、SqliteRepo、CheckpointStore(LangGraph Checkpointer 适配)。

表:tasks / checkpoints / artifacts / logs / testcases / memory_runtime / memory_edit。

边界:断点恢复依赖 checkpoints;备份/容灾留 G11;Runtime/Edit 记忆分表。

M09 Versioning Git 版本管理

职责:纳管当前项目全量代码(agents/orchestrator/skills/templates + backend/ + frontend/);main 生效、feature/* 改动、PR、git revert 回滚;支撑 F-E.7 自改代码、F-E.6 自重启。

关键类:GitService(封装 git.* Tool)、BranchManager、PRComposer、ServiceRestarter(F-E.6 软重启,健康检查失败自动 revert 回滚)。

数据流:Edit 改动→feature 分支 diff→回归过→PR→Host 确认 Merge→main→(可选)自重启前后端→Runtime 重载。

边界:远端 https://github.com/WaWaSamuel/opc_coding_studio.git 唯一真源;commit message 记 badcase/优化项满足审计;写白名单目录、黑名单(data//.env/密钥/.git)不可写;edit_git_enabled / edit_auto_restart_enabled 默认 False 保命。

M10 Self-Evolve 自迭代

M11 Observability 可观测
