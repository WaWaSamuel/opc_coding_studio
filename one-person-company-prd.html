第一部分 · 一人公司功能点全景拆解

把系统所有功能点按 6 大能力域(A 入口 / B 编排 / C 上下文 / D 可靠性 / E 自迭代 / F 基础设施)列全,每项给唯一编号 F-x.y。

总览 系统定位与架构

用 AI 驱动「拟公司化」多角色协作系统,让一人具备一家公司产能。模型被当作无状态角色函数,由 LangGraph 编排,在不同节点扮演不同角色。两套系统:Runtime(跑业务)与 Edit(改系统自身)。

│ 入口层 Gateway:飞书机器人 ║ Web 对话  →  HostCommand 归一化 │
└───────────────────────────┬──────────────────────────────┘
                            ▼
│ Web 界面(流式·日志·TODO·成本·决策) + Edit 工作流可视化(DAG)│
                Runtime 入口 / Edit 入口
                            ▼
   Runtime 编排(CEO→部长→执行→QA→Loop→验收) / Edit 编排
                            ▼

                            ▼
   Model Adapter(豆包 Ark · JSON 强约束 · 稳定前缀缓存 · 可切换)
                            ▼
   基础设施:DB(状态/日志/测试集/记忆·Runtime与Edit隔离)· 事件流 · 成本统计

第二部分 · 每个功能点的技术方案细节

与第一部分编号一一对应。按能力域组织,给出关键设计、接口/数据约定与实现要点。

A 方案 入口与交互

F-A.1 / A.2 / A.3 入口层 Gateway + 飞书 + Web

  • 入口层在编排器之外、之上,负责"渠道接入 + 协议转换 + 会话路由",不含业务逻辑。
  • 各渠道消息归一为统一 HostCommand,屏蔽渠道差异。
  • 飞书适配器:事件回调(@/私聊)+ 签名校验;回复用消息卡片 + 流式更新;need_decision 用交互按钮卡片。
  • Web 适配器:复用 React + SSE/WebSocket。
  • 身份校验来源为 Host;每对话映射 session_id 关联 task_id,支持多轮。
HostCommand = { "channel":"lark|web", "session_id":"...", "host_verified":true,
  "intent":"runtime|edit", "text":"自然语言指令", "attachments":[], "reply_to":"..." }

F-A.4 / A.6 / A.7 日志看板 / 成本仪表 / 人在环

  • 编排器关键节点推事件(role_start/thinking/tool_call/artifact/handoff/rework/error/need_decision/done),界面订阅渲染。
  • 成本仪表读流转日志的 tokens/latency_ms 聚合,按 task/角色展示。
  • need_decision:飞书卡片按钮 / Web 弹窗,决策回灌编排器继续。

F-A.5 TODO Plan 进度视图

  • 读 State.todo_plan,渲染勾选进度;回退时高亮 status=rework 项。

F-A.8 Edit 工作流可视化

  • 后端从 LangGraph 图解析 nodes/edges → 图 JSON,API GET /edit/graph?ref=main|feature。
  • 前端 React Flow 渲染 DAG;边标流转条件(pass/reject/need_rework)。
  • 改动 diff(节点/边增删改)颜色高亮;改前(main)vs 改后(feature)双图对比。
  • 点击节点查看该 Agent 提示词/Tool/Skill;本期只读理解,不做拖拽编辑。

B 方案 编排与角色

F-B.1 / B.2 双 LangGraph 状态图

  • Runtime 与 Edit 各一张 StateGraph;节点=角色,条件边=按 Artifact 字段流转。
  • 无依赖节点并行(前端/后端接口约定后、效果QA/功能QA 无依赖时)。
CEO ─重大→产品部长→PRD撰写→需求拆分→技术评审→PRD定稿─┐
CEO ─小改──────────────────────────────────────────┤
                                          开发部长(拆解+TODO)
        ┌─前端─┐
(接口约定→并行)┤     ├→效果QA ║ 功能QA →[Loop≤3]→需求验收→部长汇总→CEO
        └─后端─┘   reject/rework/fail──回退到指定上游

F-B.3 / B.4 / B.5 CEO 路由 / 部长拆解 / PM-PRD

  • CEO 输出结构化路由结果(目标部门 + 是否重大),编排器据此进图。
  • 部长产出 TODO Plan(每项 {id,desc,owner_role,status});PM 产出结构化 PRD,requirements[].acceptance 直喂需求验收。

F-B.7 Loop 宿主仿真

  • 读上游 Artifact + 验收标准,输出 {verdict:pass|reject,reason,suggestion};reject 条件边回退;max_iterations=3 超限置 need_decision 等 Host。

F-B.9 / B.10 Artifact 驱动 + 无状态隔离

  • 每节点 = 一次无状态调用,上下文 = 稳定公共层 + 角色层 + 检索注入 + 上一棒 Artifact。
  • 角色间只传结构化 Artifact JSON,status 字段驱动 LangGraph 条件边。

C 方案 上下文与记忆

F-C.2 / C.3 三层记忆 + Compact 压缩

记忆存储生命周期
工作记忆当前节点 prompt单次调用
任务记忆State + Checkpoint(DB)单 task
长期记忆DB + 检索跨任务
  • Compact:任务记忆超阈值,用一次 LLM 调用结构化摘要(保留决策/接口约定/未决问题),保留指向 DB 完整 Artifact 的可追溯指针,替换原始长文。

F-C.4 / C.5 / C.6 隔离 / 检索注入 / 缓存

  • Runtime / Edit 长期记忆物理隔离(两库 + 独立命名空间)。
  • 长期记忆按角色+任务检索 Top-K 注入,不全量塞 prompt。
  • 稳定公共层置顶利用 prompt caching。组装顺序:公共层→角色层→长期记忆 Top-K→任务记忆(必要时已 Compact)→上一棒 Artifact。

D 方案 可靠性与回退

两类重试独立,不共用计数器① 业务回退(质量/跨节点/loop_counters/超限等 Host)与 ② 节点重试(执行/原地/retry_counters/超限上报)各自独立;节点重试到上限不自动转业务回退。
维度① 业务回退 F-D.1② 节点重试 F-D.2
触发质量不达标(QA/Loop)schema/JSON/Tool/超时
判定验收类 agent(语义)编排器/Adapter(机械)
范围跨节点回退上游本节点原地重跑
计数loop_countersretry_counters
超限need_decision 等 Hosterror 上报/降级
  • F-D.3:Adapter 启用 response_format/JSON mode;解析失败回喂错误,计入 retry_counters,≤3。
  • F-D.4:LangGraph Checkpointer 落 SQLite,断点恢复。
  • F-D.5:Tool 按角色最小授权(QA 无 fs.write/code.build);结果按固定 schema 回写 State。

E 方案 自迭代与版本

F-E.1 Edit 全链路

Host"改系统"→Edit部长(定位+TODO)→提示词/编排/Skill工程师(feature分支diff)
   →回归测试官(≥95%)──劣化→回退→变更评审(提PR)→[Host确认Merge]→main生效
            异常:git revert 回滚

F-E.2 / E.3 Badcase + 每周单测

  • Badcase Agent 监听日志异常(rework频发/loop_reject/Host否决/error),结构化为用例入 DB 测试集。
  • Scheduler 每周跑回归,成功率 <95% 自动告警 Host 并触发 Edit 优化。

F-E.4 远端 GitHub 版本管理

  • 远端 https://github.com/WaWaSamuel/opc_coding_studio.git.git 为唯一真源;纳管 agents/orchestrator/skills/templates。
  • main 生效 + feature/* 改动 + PR + git revert 回滚;封装 git.* Tool;Token 走 GITHUB_TOKEN。
  • commit message 记录"解决什么 badcase/对应哪条优化"满足审计;Runtime 启动从 main 加载。

F 方案 基础设施

  • F-F.1 Model Adapter:统一 invoke(messages,schema)→Artifact;豆包配置走 ARK_* 环境变量;切换只改 Adapter;支持角色异构(路由小模型/开发大模型)。
  • F-F.2 SQLite ./data/app.db,接口抽象可换 PostgreSQL;记忆库 Runtime/Edit 隔离。
  • F-F.3 docker-compose 起前后端;不起外网监听(回调走 Host 自有部署)。
  • F-F.4 Tool调用/状态变更/Host决策留痕;密钥仅环境变量;敏感动作记录角色+时间+diff。
  • F-F.5 事件流总线统一编排器→界面/渠道推送。

第四部分 · 还欠缺的能力(Gap 分析)

当前 PRD 已覆盖主干,但以下能力尚未规划或仅浅层涉及,建议后续迭代补齐,按重要性排序。

缺口现状风险/影响建议
G1 评估体系/质量度量仅有 95% 单测通过率缺乏对"业务产物质量"的客观评分,Loop 判定依赖模型主观建独立 Eval 维度(规则+模型评分),量化各角色产出质量趋势
G2 成本/限额治理有成本仪表,但 token 不限失控任务可能烧大量 token,无熔断按任务/天设软硬限额 + 熔断;异构模型按成本路由
G3 并发与任务队列单任务串行为主多任务并发、优先级、资源争用未设计引入任务队列 + 并发上限 + 优先级调度
G4 长期记忆质量治理有读写+检索记忆会过期/冲突/污染检索,缺去重与淘汰记忆置信度、TTL、去重合并、冲突裁决机制
G5 安全与越权防护最小授权 + 审计Tool 危险操作二次确认(原优化3)暂未做;提示注入风险未评估补危险操作确认 + 输入消毒 + 沙箱执行边界
G6 可观测性深化流转日志 + 成本缺 trace 级链路追踪、告警、SLO引入结构化 trace、异常告警、关键指标看板
G7 测试集冷启动靠 Badcase Agent 积累初期无 badcase,测试集为空,回归形同虚设人工/合成种子用例冷启动测试集
G8 业务工作流落地仅空骨架 + 模板电商/资讯/业务优化无实际链路,价值未验证选一条业务流(如电商)做端到端 PoC 验证框架
G9 多轮对话与意图澄清HostCommand 单向指令指令模糊时缺澄清回合,易跑偏入口层增意图澄清子流程,必要时反问 Host

建议优先级近期(配合主干):G7 测试集冷启动、G2 成本熔断、G8 业务 PoC;中期:G1 评估体系、G4 记忆治理、G5 安全;远期:G3 并发、G6 可观测深化、G10 门禁、G11 容灾。

5.1 技术模块详细设计

系统拆为 12 个核心模块,统一编号 M-编号。每个模块标注:职责、关键类/接口、依赖、输入输出、失败与边界处理。

M01 Gateway 入口层

职责:多渠道接入(飞书/Web)、协议归一为 HostCommand、会话路由、身份(Host)校验、意图初判(runtime/edit)。

关键类:ChannelAdapter(抽象)→ LarkAdapter / WebAdapter;SessionRouter;HostCommand(DTO);IntentClassifier(轻量规则+小模型)。

数据流:渠道事件 →(签名校验/去重)→ 归一 HostCommand →(session_id↔task_id 映射)→ 投递 Orchestrator。

边界:飞书事件需做 x-lark-signature 校验 + event_id 幂等去重;非 Host 来源直接拒绝;意图模糊触发澄清子流程(G9 预留)。

失败:渠道回调失败→本地重投队列;归一失败→回渠道提示格式错误,不进编排。

M02 Orchestrator 编排内核

职责:持有 Runtime / Edit 两张 StateGraph;按 Artifact 字段走条件边;管理 Loop/并行/回退;驱动节点=角色调用。

关键类:GraphBuilder(装配节点/边)、RuntimeGraph/EditGraph、NodeRunner(包裹一次无状态调用)、EdgeRouter(读 status→下一跳)、CompanyState(TypedDict,含 transition 字段记录"为何流转至此",借 Claude Code)。

数据流:State 入图 → NodeRunner 组装上下文 → ModelAdapter 调用 → 回写 Artifact → EdgeRouter 判定(pass/reject/need_rework/error)→ 下一节点 / 回退 / need_decision。

边界:并行节点(前后端)需在汇合节点 join;条件边必须覆盖所有 status 取值,缺省落 error 兜底节点。

失败:节点异常→交 M07 重试;超 max_iterations→置 need_decision 等 Host;图无可达边→error 上报。

M03 Role / Agent 注册层

职责:角色定义(提示词三层 + 可用 Tool/Skill + 模型档位 + 输出 schema)统一注册,供编排器按 current_role 取用。

关键类:RoleRegistry、RoleSpec{ role_id, model_tier, prompt_layers, tools[], output_schema }、PromptComposer(三层拼装)。

数据流:角色定义从 main 分支 YAML/JSON 加载 → RoleRegistry 内存索引 → NodeRunner 取 RoleSpec → PromptComposer 组装。

边界:提示词三层(common/role/task)解耦,改 common 影响全局须经 Edit 回归;model_tier 决定走小/大模型。

M04 Model Adapter 模型适配

职责:屏蔽模型差异,统一 invoke(messages, schema, tier) → Artifact;强制 JSON 输出;统计 tokens/latency;支持角色异构路由。

关键类:ModelAdapter(抽象)→ ArkAdapter(豆包,OpenAI 兼容);InvokeResult{content, tokens, latency_ms};SchemaEnforcer(response_format)。

数据流:messages →(注入 response_format)→ Ark API → 解析 JSON →(失败回 M07 自修复)→ 结构化 Artifact。

边界:密钥仅 ARK_* 环境变量;切模型只换 Adapter 实现;tier=small/large 映射不同 endpoint。

失败:JSON 解析失败→把错误回喂模型自修复(≤3);API 5xx→指数退避重试(计入 retry_counters)。

M05 Context & Memory 上下文记忆

职责:三层记忆(工作/任务/长期)读写;三级上下文流水线压缩;Runtime/Edit 物理隔离;检索 Top-K 注入;稳定前缀缓存。

关键类:MemoryManager、WorkingMem/TaskMem/LongTermMem、ResultBudgeter(Tier1)、MicroCompactor(Tier2)、Compactor(Tier3 LLM 摘要+指针)、Retriever(向量/关键词 Top-K)。

三级流水线(借 Claude Code,每次调用前依次执行):① 结果预算:大产出落 DB,prompt 内留 ≤2KB 预览 + 指针(按需 fetch_artifact 取回);② 微压缩:按来源选择性清旧结果,以缓存过期为触发;③ 全量压缩:超阈值时一次 LLM 结构化九段摘要(含 Host 原话逐字引用),替换原文但保留可追溯指针。

数据流(缓存边界):公共层 → 角色层 → 长期记忆 Top-K ┊CACHE_BOUNDARY┊ 任务记忆(必要时已压缩)→ 上一棒 Artifact;边界前稳定,命中 prompt caching。

边界:Compact 保留指向 DB 完整 Artifact 的指针(可追溯,非截断);Runtime/Edit 两库两命名空间。

失败:检索为空→降级仅用任务记忆;Compact 连续失败计入熔断(M07),超限保守保留原文并告警。

M06 Tool / Skill 执行层

职责:统一工具注册与最小授权调用(fs/code.build/git.*/test.run/lark.* 等);结果按固定 schema 回写 State。

关键类:ToolRegistry、Tool{name, scopes, schema, handler}、ToolInvoker(鉴权+执行+留痕)、DangerDetector(规则/AST 危险动作检测)。

四层授权流(借 Claude Code,补 G5):① 配置规则(角色↔Tool 白名单,最快)→ ② Hook(可扩展校验)→ ③ 规则/AST 危险动作检测(fs 越界写、git reset --hard、删表等)→ ④ Host 二次确认(兜底)。危险动作"警告但不阻断,需显式确认"。

数据流:角色请求 Tool → ToolInvoker 四层校验 → 执行 → 结构化结果 → 审计留痕 → 回 State。

边界:QA 角色无 fs.write/code.build;危险操作二次确认本期以 hook 占位,门禁后启用;git.* 走 GITHUB_TOKEN。

M07 Reliability 可靠性(双重试 + 全局熔断)

职责:实现两类独立机制:① 业务回退 loop_counters(质量/跨节点/超限等 Host);② 节点重试 retry_counters(schema/JSON/Tool/超时/原地)。再叠加全局熔断常量表覆盖所有自动恢复路径。

关键类:LoopController(业务回退)、RetryController(节点重试)、SelfRepair(JSON 回喂修复)、CircuitBreakers(全局熔断)。

全局熔断常量(借 Claude Code circuit breaker,防连环失败烧 token):

MAX_COMPACT_FAILURES   = 3   # 连续压缩失败→停压缩报 Host
MAX_TOKEN_REFRESH      = 3   # 鉴权刷新失败上限
MAX_API_OVERLOAD_RETRY = 3   # 模型 5xx/限流重试
MAX_POLL_FAILURES      = 5   # 渠道轮询失败
MAX_TASK_TOKENS / DAY  = 软硬双限      # 成本熔断(补 G2)

硬约束:节点重试到上限 不自动转 业务回退;两个计数器互不共用;每条自动恢复路径必须有对应熔断上限。

失败:loop 超 3→need_decision;retry 超 3→error 上报/降级;熔断触发→停该恢复路径并告警 Host。

M08 Persistence 持久化

职责:SQLite(./data/app.db)落 State/Checkpoint/Artifact/日志/测试集/记忆;接口抽象可换 PostgreSQL。

关键类:Repository(抽象)、SqliteRepo、CheckpointStore(LangGraph Checkpointer 适配)。

表:tasks / checkpoints / artifacts / logs / testcases / memory_runtime / memory_edit。

边界:断点恢复依赖 checkpoints;备份/容灾留 G11;Runtime/Edit 记忆分表。

M09 Versioning Git 版本管理

职责:纳管 agents/orchestrator/skills/templates;main 生效、feature/* 改动、PR、git revert 回滚。

关键类:GitService(封装 git.* Tool)、BranchManager、PRComposer。

数据流:Edit 改动→feature 分支 diff→回归过→PR→Host 确认 Merge→main→Runtime 重载。

边界:远端 https://github.com/WaWaSamuel/opc_coding_studio.git.git 唯一真源;commit message 记 badcase/优化项满足审计。

M10 Self-Evolve 自迭代

职责:Badcase Agent 监听日志异常并结构化为用例;Scheduler 每周回归;<95% 告警 Host + 触发 Edit。

关键类:BadcaseCollector、TestSuiteRunner、Scheduler(APScheduler)、EvalReporter。

边界:测试集冷启动需种子用例(G7);评估目前仅通过率,质量评分留 G1。

M11 Observability 可观测

职责:统一事件流总线(role_start/thinking/tool_call/artifact/handoff/rework/error/need_decision/done);流转日志 + 成本聚合。

关键类:EventBus(内存 pub/sub + 落库)、LogEntry、CostAggregator。

数据流:编排器发事件 → EventBus → ①落 logs 表 ②推 SSE/WS 到界面/渠道。

边界:trace 级链路/告警/SLO 留 G6。

M12 Web UI 交互界面

职责:对话下指令、流式反馈、TODO 进度、成本仪表、need_decision 决策、Edit 工作流可视化(React Flow + diff 双图)。

关键类:ChatPanel、EventStream(SSE/WS hook)、TodoView、CostPanel、DecisionModal、GraphView(React Flow)。

边界:Edit 可视化本期只读理解,不做拖拽编辑;决策回灌编排器继续。
